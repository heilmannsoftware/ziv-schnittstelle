Einführung

Das ZIV Datenmodell 4.0 definiert ein einheitliches Datenbankschema für den Austausch von Kehrbezirksdaten zwischen verschiedenen proprietären Softwaresystemen der Schornsteinfegerbranche. Es ermöglicht den verlustfreien Transfer aller relevanten Daten beim Wechsel der Verwaltungssoftware oder bei der Übergabe eines Kehrbezirks.

Wofür wird das Modell gebraucht?

  • Softwarewechsel: Ein Schornsteinfeger wechselt von Programm A zu Programm B — alle Daten müssen übertragen werden
  • Kehrbezirksübergabe: Ein Bezirk wird an einen neuen bevollmächtigten Bezirksschornsteinfeger übergeben
  • Archivierung: Erstellen einer standardisierten Datensicherung des Kehrbezirks

Grundprinzip

Der Datenaustausch erfolgt über eine SQLite-Datenbank. Das exportierende System schreibt alle Daten in eine .db-Datei nach dem definierten Schema. Das importierende System liest diese Datei und überführt die Daten in seine eigene Struktur.

Exportierendes System Importierendes System Programm A Programm B ┌─ Eigene Datenbank ┌─ Eigene Datenbank ┌─ Mapping auf ZIV-Schema ┌─ Mapping von ZIV-Schema │ │ └────> kehrbezirk_export.db ────>└──> Import & Zuordnung (SQLite, ZIV 4.0)
ⓘ
Warum SQLite?

SQLite ist plattformunabhängig, benötigt keinen Server, ist in jeder Programmiersprache verfügbar und die gesamte Datenbank ist eine einzelne Datei. Das macht es ideal für den Offline-Datenaustausch.

Schnelleinstieg

Um das Datenmodell in Ihrer Software zu integrieren, benötigen Sie im Wesentlichen drei Schritte:

1

Schema erstellen

Führen Sie schema.sql und stammdaten_enums.sql gegen eine neue SQLite-Datenbank aus. Damit haben Sie alle 89 Tabellen und 118 Enum-Wertetabellen.

2

Daten mappen

Erstellen Sie ein Mapping zwischen Ihren internen Datenstrukturen und den ZIV-Tabellen. Die Dokumentation beschreibt jedes Feld mit Datentyp, Constraints und fachlicher Bedeutung.

3

Import/Export implementieren

Implementieren Sie die Export-Logik (eigene DB → ZIV-SQLite) und Import-Logik (ZIV-SQLite → eigene DB). Beachten Sie dabei die Einfügereihenfolge der Tabellen.

-- Schnelltest: Schema erstellen und validieren
sqlite3 export.db < schema.sql
sqlite3 export.db < stammdaten_enums.sql
sqlite3 export.db "SELECT count(*) FROM sqlite_master WHERE type='table';"
-- Erwartetes Ergebnis: 208 (89 Datentabellen + ~118 Enum-Tabellen + sqlite_sequence)

Standort-Hierarchie

Das Datenmodell ist als strenge Baumstruktur aufgebaut. Jedes Gebäude, jede Wohnung, jeder Raum und jede Anlage hängt an einem definierten Elternobjekt. Der Baum beginnt beim Ort und endet beim einzelnen Raum.

Ort └─ Strasse n Strassen pro Ort └─ Grundstueck n Grundstücke pro Strasse ┌─ Grundstueck_Eigentuemer n:m über Junction-Table └─ Gebaeude_Gebaeudeteil n Gebäude pro Grundstück ┌─ Gebaeude_Eigentuemer n:m über Junction-Table ┌─ Abgasanlage n Abgasanlagen pro Gebäude ┌─ Lueftungsanlage n Lüftungsanlagen pro Gebäude ┌─ Dunstabzuganlage_Leitung n Dunstabzugsleitungen pro Gebäude └─ Nutzungseinheit n Nutzungseinheiten pro Gebäude ┌─ Brennstoffversorgungsanlage ┌─ Raeucheranlage └─ Wohnung n Wohnungen pro Nutzungseinheit └─ Raum n Räume pro Wohnung ┌─ Feuerstaette n Feuerstätten pro Raum └─ Waermepumpe n Wärmepumpen pro Raum
⚠
Wichtig: Einfügereihenfolge beachten!

Beim Import müssen Eltern-Datensätze vor ihren Kindern eingefügt werden, da Foreign Keys aktiv sind. Ort → Strasse → Grundstueck → Gebaeude → Nutzungseinheit → Wohnung → Raum → Feuerstaette.

Eigentümer (n:m-Beziehung)

Grundstücke und Gebäude können mehrere Eigentümer haben. Dies wird über Junction-Tables abgebildet:

-- Eigentümer eines Grundstücks abfragen
SELECT k.Name, k.Vorname, ge.Anteil, ge.Sprecher
FROM Grundstueck_Eigentuemer ge
JOIN Kommunikation k ON k.ID_Kommunikation = ge.ID_Kommunikation
WHERE ge.ID_Grundstueck = 42;

-- Struktur der Junction-Table:
-- ID_Grundstueck_Eigentuemer  (PK)
-- ID_Grundstueck              (FK → Grundstueck)
-- ID_Kommunikation            (FK → Kommunikation, = Person/Firma)
-- Anteil                      (REAL, z.B. 0.5 für 50%)
-- Sprecher                    (BOOL, Sprecher der Eigentümergemeinschaft)

Anlagen & Feuerstätten

Anlagen sind technische Einrichtungen in einem Gebäude, die vom Schornsteinfeger geprüft und gewartet werden. Sie sind die zentralen Fachobjekte des Datenmodells.

Abgasanlage im Detail

Eine Abgasanlage (Schornstein) hat eine hierarchische Unterstruktur:

Abgasanlage z.B. ein Schornstein ┌─ Abschnitt (Sortiernummer) z.B. 3 Abschnitte übereinander │ └─ Schicht Innenschicht, Außenschicht etc. ┌─ Muendung (1:1, UNIQUE) Genau eine Mündung pro Abgasanlage ┌─ Reinigungsart n Reinigungsarten pro Abgasanlage └─ Feuerstaette (n:1) n Feuerstätten angeschlossen

Dunstabzug-Hierarchie

Dunstabzuganlage_Leitung Dunstabzugsschacht im Gebäude ┌─ Abschnitt_Dunstabzugsleitung Abschnitte mit Sortiernummer ┌─ Ventilator_Dunstabzug Ventilatoren └─ Dunstabzugsanlage_Haube Hauben (in einem Raum) └─ Dunstabzugsanlage_Feuerstaette Feuerstätten an der Haube

Spezialisierungsmuster

Feuerstätten verwenden ein Table-per-Type-Muster. Es gibt eine Basistabelle Feuerstaette mit gemeinsamen Feldern und spezialisierte Tabellen für jeden Brennstofftyp mit einer 1:1-Beziehung.

Feuerstaette (Basistabelle: Raum, Abgasanlage, gemeinsame Felder) ┌─ Gasfeuerstaette 1:1 → spezifische Gas-Felder │ └─ Mess_Pruefergebnisse_Gas... n Messergebnisse ┌─ Oelfeuerstaette 1:1 → spezifische Öl-Felder │ └─ Mess_Pruefergebnisse_Oel... n Messergebnisse ┌─ Feste_Brennstoff_Feuerstaette 1:1 → Holz/Kohle/Pellets │ ┌─ Mess_Pruefergebnisse_..._HK n Handkehrmessungen │ └─ Mess_Pruefergebnisse_..._ERF n ERF-Messungen └─ Sonderfeuerstaetten 1:1 → BHKW, Brennstoffzellen etc. └─ Mess_Pruefergebnisse_Sonder... n Messergebnisse 
-- Beispiel: Gas-Feuerstätte mit Messergebnissen abfragen
SELECT
    f.ID_Feuerstaette,
    g.Nennwaermeleistung,
    g.Hersteller,
    m.Abgastemperatur,
    m.CO_Wert,
    m.Datum_Messung
FROM Feuerstaette f
JOIN Gasfeuerstaette g ON g.ID_Feuerstaette = f.ID_Feuerstaette
LEFT JOIN Mess_Pruefergebnisse_Gasfeuerstaette m
    ON m.ID_Gasfeuerstaette = g.ID_Gasfeuerstaette
WHERE f.ID_Raum = 123;
💡
Tipp für Entwickler

Beim Import einer Feuerstätte müssen Sie immer zuerst den Basisdatensatz in Feuerstaette anlegen und dann den spezialisierten Datensatz (z.B. Gasfeuerstaette) mit Verweis auf die neue ID.

Kehrbuch (Snapshot-Prinzip)

Das Kehrbuch ist das zentrale Nachweisinstrument des Schornsteinfegers. Es dokumentiert für einen bestimmten Zeitraum (Geschäftsjahr) den Zustand aller Anlagen und durchgeführten Tätigkeiten.

ⓘ
Warum Snapshot?

Stammdaten (Eigentümer, Adressen) können sich ändern. Das Kehrbuch speichert den Zustand zum Zeitpunkt der Eintragung. Daher gibt es eigene Kehrbuch-Tabellen, die Stammdaten-Referenzen einfrieren.

Kehrbuch (pro Kehrbezirk + Zeitraum) └─ Kehrbuch_Gebaeude Gebäude-Snapshot (+ Eigentümer, Verwalter, Mieter) ┌─ Kehrbuch_Gebaeude_Eigentuemer Eigentümer zum Kehrbuchzeitpunkt ┌─ Kehrbuch_Nutzungseinheit Nutzungseinheit-Snapshot │ ┌─ Kehrbuch_Abnahme Referenz → Abnahme │ └─ Kehrbuch_Anlassbez._Ueberpruefung └─ Kehrbuch_Anlage Polymorphe Anlagen-Referenz └─ Kehrbuch_Taetigkeit Durchgeführte Arbeiten └─ Mangel Festgestellte Mängel

Kehrbuch_Anlage: Polymorphe Referenz

Die Tabelle Kehrbuch_Anlage hat 8 optionale Foreign Keys, von denen genau einer gesetzt sein sollte. So wird jede beliebige Anlage im Kehrbuch referenziert:

-- Kehrbuch_Anlage: Genau ein FK ist gesetzt
ID_Abgasanlage                 -- Schornstein/Abgasanlage
ID_Feuerstaette                -- Feuerstätte
ID_Raumluftoeffnung            -- Raumöffnung
ID_Dunstabzuganlage_Leitung    -- Dunstabzugsschacht
ID_Lueftungsanlage             -- Lüftungsanlage
ID_Raumlueftung                -- Raumlüftung
ID_Raeucheranlage              -- Räucheranlage
ID_Brennstoffversorgungsanlage -- Brennstoffversorgung

Enums & Stammdaten

Das Modell verwendet 118 Enum-Tabellen als Wertelisten. Diese sind in stammdaten_enums.sql definiert und stellen sicher, dass nur gültige Werte verwendet werden.

Aufbau einer Enum-Tabelle

CREATE TABLE IF NOT EXISTS Enum_AbgasanlageArt (
    Wert TEXT PRIMARY KEY,
    Beschreibung TEXT
);

INSERT INTO Enum_AbgasanlageArt (Wert, Beschreibung) VALUES
  ('SN', 'Schornstein | Erläuterung: ...'),
  ('AL', 'Abgasleitung (ohne Schacht) | ...'),
  ('VS', 'Verbindungsstück | ...');

Verwendung in der Datentabelle

Die Datentabellen verwenden CHECK-Constraints, die auf die gültigen Enum-Werte verweisen:

CREATE TABLE Abgasanlage (
    ...
    Art TEXT CHECK(Art IN ('AB','AL','AT','AX','LA',...)),
    ...
);
⚠
CHECK-Constraints sind aktiv!

SQLite prüft die CHECK-Constraints beim INSERT. Stellen Sie sicher, dass Ihre Werte exakt mit den Enum-Werten übereinstimmen (case-sensitive!). Eine Abfrage der Enum-Tabelle vor dem Export hilft, Zuordnungsfehler zu vermeiden.

Import-Reihenfolge

Da Foreign Keys aktiviert sind (PRAGMA foreign_keys = ON), müssen Datensätze in der richtigen Reihenfolge eingefügt werden. Die schema.sql ist bereits topologisch sortiert — die Tabellenreihenfolge in der Datei ist eine gültige Einfügereihenfolge.

Empfohlene Reihenfolge (vereinfacht)

2Kommunikation, VertreterStammdaten ohne FK-Abhängigkeiten untereinander
3Kehrbezirk, Mitarbeiter, Fremdfirma, UABVerwaltung (referenziert Kommunikation/Vertreter)

Pragmatischer Ansatz

Sie können PRAGMA foreign_keys = OFF setzen, alle Daten in beliebiger Reihenfolge laden, und danach mit PRAGMA foreign_key_check die Integrität prüfen. Das ist oft einfacher als eine exakte Sortierung.

Export erstellen

Als Softwarehersteller implementieren Sie den Export Ihrer internen Daten in das ZIV-Format. Hier die empfohlenen Schritte:

Schritt-für-Schritt

1

Leere Datenbank erzeugen

Erstellen Sie eine neue SQLite-Datei und führen Sie schema.sql sowie stammdaten_enums.sql aus.

2

Meta-Tabelle befüllen

Schreiben Sie einen Eintrag in Meta mit Exportdatum, Softwarename, Version und Modellversion (4.0).

3

ID-Mapping aufbauen

Ihre internen IDs weichen vermutlich von den ZIV-IDs ab. Bauen Sie während des Exports eine Mapping-Tabelle auf: interne_id → ziv_id. Das brauchen Sie, um Foreign Keys korrekt zu setzen.

4

Daten schreiben (top-down)

Schreiben Sie die Daten in der Import-Reihenfolge. Nutzen Sie AUTOINCREMENT-IDs — SQLite vergibt diese automatisch, wenn Sie die ID-Spalte beim INSERT weglassen.

5

Validieren

Prüfen Sie die Datei mit PRAGMA integrity_check und PRAGMA foreign_key_check.

-- Meta-Eintrag beim Export
INSERT INTO Meta (Abgabe, Version_Modell, Software, Version_Software, Geschaeftsjahr)
VALUES (
    '2026-03-09T14:30:00',  -- Exportzeitpunkt
    '4.0',                  -- ZIV-Modellversion
    'MeineProgramm Pro',    -- Name Ihrer Software
    '3.2.1',                -- Ihre Softwareversion
    2025                    -- Geschäftsjahr
);

Validierung

Nach dem Befüllen der Export-Datenbank sollten Sie mehrere Prüfungen durchführen:

1. Strukturelle Integrität

-- Allgemeine Integritätsprüfung
PRAGMA integrity_check;
-- Erwartet: "ok"

-- Foreign-Key-Prüfung (findet verwaiste Referenzen)
PRAGMA foreign_key_check;
-- Erwartet: keine Ergebnisse (= keine Verletzungen)

2. Vollständigkeitsprüfungen

-- Gibt es mindestens einen Kehrbezirk?
SELECT count(*) FROM Kehrbezirk;

-- Haben alle Gebäude ein Grundstück?
SELECT g.ID_Gebaeude_Gebaeudeteil
FROM Gebaeude_Gebaeudeteil g
LEFT JOIN Grundstueck gr ON gr.ID_Grundstueck = g.ID_Grundstueck
WHERE gr.ID_Grundstueck IS NULL;
-- Erwartet: keine Ergebnisse

-- Haben alle Feuerstätten genau eine Spezialisierung?
SELECT f.ID_Feuerstaette,
    (g.ID_Gasfeuerstaette IS NOT NULL) +
    (o.ID_Oelfeuerstaette IS NOT NULL) +
    (fb.ID_Feste_Brennstoff_Feuerstaette IS NOT NULL) +
    (s.ID_Sonderfeuerstaetten IS NOT NULL) AS anz
FROM Feuerstaette f
LEFT JOIN Gasfeuerstaette g ON g.ID_Feuerstaette = f.ID_Feuerstaette
LEFT JOIN Oelfeuerstaette o ON o.ID_Feuerstaette = f.ID_Feuerstaette
LEFT JOIN Feste_Brennstoff_Feuerstaette fb ON fb.ID_Feuerstaette = f.ID_Feuerstaette
LEFT JOIN Sonderfeuerstaetten s ON s.ID_Feuerstaette = f.ID_Feuerstaette
WHERE anz != 1;
-- Erwartet: keine Ergebnisse (jede Feuerstätte hat genau 1 Typ)

3. Validierung mit Python

import sqlite3

conn = sqlite3.connect('export.db')
conn.execute('PRAGMA foreign_keys = ON')

# Strukturcheck
result = conn.execute('PRAGMA integrity_check').fetchone()
assert result[0] == 'ok', f'Integritätsfehler: {result}'

# FK-Check
violations = conn.execute('PRAGMA foreign_key_check').fetchall()
assert len(violations) == 0, f'FK-Verletzungen: {violations}'

# Tabellenanzahl prüfen
tables = conn.execute(
    "SELECT count(*) FROM sqlite_master WHERE type='table'"
).fetchone()[0]
print(f'Tabellen: {tables}')  # Erwartet: ~208

conn.close()

Art-Klassifikation der Felder

Jedes Feld im Datenmodell ist mit einer Art-Kennzeichnung versehen, die Ihnen beim Mapping hilft. Diese Kennzeichnung gibt an, woher das Feld stammt und ob es für den Feuerstättenbescheid relevant ist.

ChipBedeutungAnzahlHinweis für Entwickler
QS Qualitätssicherung / eigene Daten ~350 Felder Interne Verwaltungsdaten, nicht auf dem Feuerstättenbescheid
19K § 19 Kommunikation (Kundenrelevant) ~210 Felder Felder, die auf dem Feuerstättenbescheid für den Kunden erscheinen
19B § 19 Behörde (Meldepflichtig) ~590 Felder Felder, die an die Aufsichtsbehörde gemeldet werden
NEIN Nicht im Austauschformat vorgesehen ~21 Felder Können beim Export leer gelassen werden
💡
Priorisierung beim Mapping

Beginnen Sie mit den 19K und 19B Feldern — diese sind gesetzlich vorgeschrieben. QS-Felder sind ergänzende Qualitätsdaten, die den Austausch verbessern.

Zusätzlich gibt es Varianten wie QS MGH (Mehrgebäudehaus-spezifisch) oder 19K MGH. Die Zuordnung finden Sie in der interaktiven Dokumentation in der Spalte “Art”.

Beispiel: Vollständiger Import (Python)

Dieses Beispiel zeigt, wie Sie eine ZIV-Export-Datei einlesen und die Standort-Hierarchie in Ihre eigene Datenstruktur überführen.

import sqlite3

def import_ziv(ziv_db_path):
    """Importiert eine ZIV 4.0 Exportdatei."""
    conn = sqlite3.connect(ziv_db_path)
    conn.row_factory = sqlite3.Row
    conn.execute('PRAGMA foreign_keys = ON')

    # 1. Meta prüfen
    meta = conn.execute('SELECT * FROM Meta').fetchone()
    print(f"Import von: {meta['Software']} {meta['Version_Software']}")
    print(f"Modellversion: {meta['Version_Modell']}")

    # 2. Standort-Hierarchie einlesen
    orte = conn.execute('SELECT * FROM Ort').fetchall()
    for ort in orte:
        # In eigene DB überführen...
        strassen = conn.execute(
            'SELECT * FROM Strasse WHERE ID_Ort = ?',
            (ort['ID_Ort'],)
        ).fetchall()

        for strasse in strassen:
            grundstuecke = conn.execute(
                'SELECT * FROM Grundstueck WHERE ID_Strasse = ?',
                (strasse['ID_Strasse'],)
            ).fetchall()

            for gs in grundstuecke:
                # Eigentümer auslesen (n:m)
                eigentuemer = conn.execute("""
                    SELECT k.*, ge.Anteil, ge.Sprecher
                    FROM Grundstueck_Eigentuemer ge
                    JOIN Kommunikation k ON k.ID_Kommunikation = ge.ID_Kommunikation
                    WHERE ge.ID_Grundstueck = ?
                """, (gs['ID_Grundstueck'],)).fetchall()

                # Gebäude auslesen
                gebaeude = conn.execute(
                    'SELECT * FROM Gebaeude_Gebaeudeteil WHERE ID_Grundstueck = ?',
                    (gs['ID_Grundstueck'],)
                ).fetchall()

                for geb in gebaeude:
                    # Nutzungseinheiten → Wohnungen → Räume → Feuerstätten
                    import_gebaeude_details(conn, geb)

    conn.close()

Beispiel: Export erstellen (Python)

import sqlite3
import os

def export_ziv(output_path, eigene_db):
    """Erstellt eine ZIV 4.0 Exportdatei."""

    # 1. Neue DB mit Schema erstellen
    if os.path.exists(output_path):
        os.remove(output_path)

    conn = sqlite3.connect(output_path)
    conn.execute('PRAGMA foreign_keys = OFF')  # Erst am Ende prüfen

    # Schema laden
    with open('schema.sql', 'r', encoding='utf-8') as f:
        conn.executescript(f.read())
    with open('stammdaten_enums.sql', 'r', encoding='utf-8') as f:
        conn.executescript(f.read())

    # 2. Meta
    conn.execute("""
        INSERT INTO Meta (Abgabe, Version_Modell, Software, Version_Software, Geschaeftsjahr)
        VALUES (datetime('now'), '4.0', 'MeinSoftware', '2.0', 2025)
    """)

    # 3. ID-Mapping: interne ID → neue ZIV-ID
    id_map = {}  # {'Tabelle:interne_id': ziv_id}

    # 4. Kommunikation exportieren (keine FK-Abhängigkeiten)
    for person in eigene_db.get_alle_personen():
        cursor = conn.execute("""
            INSERT INTO Kommunikation (Name, Vorname, Strasse, PLZ, Ort, Land)
            VALUES (?, ?, ?, ?, ?, ?)
        """, (person.name, person.vorname, person.strasse,
            person.plz, person.ort, 'DE'))
        id_map[f'Kommunikation:{person.id}'] = cursor.lastrowid

    # 5. Grundstücke, Gebäude etc. analog...
    # (immer id_map verwenden für FK-Werte)

    # 6. Validierung
    conn.execute('PRAGMA foreign_keys = ON')
    violations = conn.execute('PRAGMA foreign_key_check').fetchall()
    if violations:
        print(f"WARNUNG: {len(violations)} FK-Verletzungen!")
        for v in violations:
            print(f"  Tabelle: {v[0]}, Row: {v[1]}, "
                  f"Referenz: {v[2]}, FK-Index: {v[3]}")
    else:
        print("Export erfolgreich validiert!")

    conn.commit()
    conn.close()
    print(f"Export geschrieben: {output_path}")

Typische Stolperfallen
  • DATE-Format: Verwenden Sie YYYY-MM-DD (ISO 8601). Nicht DD.MM.YYYY!
  • BOOL-Werte: Verwenden Sie 0 und 1 (nicht 'true'/'false')
  • Leere Strings vs. NULL: Optionale Felder sollten NULL sein, nicht ''
  • Encoding: SQLite speichert Text als UTF-8. Stellen Sie sicher, dass Umlaute korrekt sind
  • Dezimaltrennzeichen: Punkt, nicht Komma (12.5 nicht 12,5)

FAQ

Muss ich alle 89 Tabellen befüllen?

Nein. Befüllen Sie nur die Tabellen, für die Ihr System Daten hat. Leere Tabellen sind kein Problem. Die Meta-Tabelle ist jedoch Pflicht.

Kann ich eigene IDs verwenden?

Ja, Sie können die AUTOINCREMENT-IDs selbst vergeben, solange sie eindeutig sind und die FK-Beziehungen stimmen. Alternativ lassen Sie SQLite die IDs vergeben und merken sich das Mapping.

Was passiert mit Feldern, die mein System nicht kennt?

Lassen Sie diese als NULL (sofern kein NOT NULL). Das importierende System muss damit umgehen können.

Wie gehe ich mit Änderungen am Modell um?

Die Modellversion steht in Meta.Version_Modell. Prüfen Sie diese beim Import und passen Sie Ihr Mapping entsprechend an. Abwärtskompatible Änderungen (neue Spalten) sind unkritisch.

Muss ich die Enum-Tabellen mit exportieren?

Ja. Führen Sie stammdaten_enums.sql beim Erstellen der Exportdatei aus. Die Enum-Tabellen dienen als Referenz und werden für die CHECK-Constraints benötigt.

Was bedeuten die DEPRECATED-Tabellen?

Die Tabellen Belegungsplan und Bericht sind als veraltet markiert. Sie existieren noch für die Abwärtskompatibilität, sollten aber nicht mehr aktiv befüllt werden. Verwenden Sie stattdessen die Dokument-Tabelle.

Wo finde ich die Bedeutung eines einzelnen Feldes?

Öffnen Sie die interaktive Dokumentation. Dort können Sie nach Tabellen- und Feldnamen suchen und sehen Datentyp, Constraints, FK-Beziehungen, Enum-Referenzen und die fachliche Beschreibung.

Kann ich die Datenbank auch mit anderen Sprachen als Python lesen?

Ja. SQLite-Bibliotheken gibt es für praktisch jede Sprache: C#/.NET (Microsoft.Data.Sqlite), Java (sqlite-jdbc), PHP (PDO::sqlite), Node.js (better-sqlite3), Delphi, Go, Rust, etc.
